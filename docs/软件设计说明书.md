# AgentFlow-Eval 软件设计说明书

## 1. 引言

### 1.1 项目背景

AgentFlow-Eval 是一个面向企业 AI 应用开发者的 Agent 自动化评测工作台。随着大语言模型（LLM）在企业场景中的广泛应用，如何系统性地评估 AI Agent 在执行业务任务时的表现成为一个关键需求。本项目提供了一套完整的评测流水线：上传业务测试用例 -> 自动运行 Agent -> 可视化展示执行链路 -> 生成评测报告。

### 1.2 项目目标

- 提供可视化 Agent 执行链路追踪（基于 ReactFlow 的 DAG 图）
- 支持自定义业务指标打分（基于 LLM-as-Judge 的多维度评分引擎）
- 支持异步任务队列（基于 Celery + Redis）
- 通过 CSV/JSON 文件批量导入测试用例
- 生成结构化的评测报告，包含各维度分数分布和详细扣分说明

### 3.3 核心算法

#### a. OpenAIReActRunner（多轮 ReAct 循环执行器）

**文件**: `backend/app/core/agent_runner/openai_runner.py`

**设计思路**:
实现完整的 ReAct（Reasoning + Acting）循环，让 LLM 能够迭代推理、调用工具、观察结果，直到得出最终答案。

**执行流程**:

```
用户 Query
   │
   ▼
┌──────────────────────────────────────────────────────────────┐
│                    ReAct Loop (max_iterations=5)              │
│                                                              │
│  ┌─────────┐     ┌──────────┐     ┌───────────┐             │
│  │ Thought  │────→│  Action  │────→│Observation│             │
│  │ (推理)   │     │ (行动)   │     │ (观察)    │             │
│  └─────────┘     └──────────┘     └───────────┘             │
│       │               │                │                    │
│       │        ┌──────┴───────┐        │                    │
│       │        │              │        │                    │
│       │   ┌──────────┐ ┌───────────┐  │                    │
│       │   │工具函数  │ │最终答案   │  │                    │
│       │   │执行      │ │final_answer│  │                    │
│       │   └──────────┘ └─────┬─────┘  │                    │
│       │                      │        │                    │
│       └──────────────────────┘────────┘                    │
│                              │                             │
│                              ▼                             │
│                        最终答案                             │
└──────────────────────────────────────────────────────────────┘
```

**两种执行模式**:

1. **函数调用模式**（默认）：使用 OpenAI 原生 tool_calls API。LLM 返回结构化工具调用指令，系统执行对应函数并将结果作为 tool response 消息送回。

2. **文本解析模式**（回退）：当 LLM 未使用函数调用格式时，解析其文本输出中的 Thought/Action/Action Input 标签。支持中英文标签。

**工具执行**:

```python
class ToolDefinition:
    name: str           # 工具名
    description: str    # 描述
    parameters: dict    # JSON Schema
    fn: Callable | None # 可选的可调用实现
```

- 当 `fn` 提供时：直接调用 `fn(**args)` 并返回结果字符串
- 内置模拟：`web_search` 和 `calculator` 两个默认实现
- `calculator` 使用 `ast.parse` 安全计算数学表达式

**返回结构**:

```python
{
    "steps": [
        {"iteration": 0, "thought": "分析...", "action": "search", "action_input": "...", "observation": "...", "tokens": 50},
        {"iteration": 1, "thought": "完成", "action": "final_answer", "action_input": "答案是...", "observation": "", "tokens": 30}
    ],
    "total_tokens": 80,
    "iterations": 2,
    "final_answer": "答案是...",
    "status": "success" | "failed" | "max_iterations_reached",
    "response_time_ms": 1234
}
```

#### b. LLMJudge（两阶段混合评分引擎）

**文件**: `backend/app/core/judge_engine/llm_judge.py`

**设计思路**:
创新性地将规则基评分与 LLM 深度语义分析相结合，实现两阶段混合评分流水线。第一阶段运行无 LLM 依赖的快速规则评分，第二阶段启用 LLM 进行语义级精化。

**评分流水线**:

```
evaluate(trace_steps, expected_output, expected_tools?)
                       │
                       ▼
┌────────────────────────────────────┐
│      Stage 1: Rule-based          │ ← 始终运行，零外部依赖
│      Pre-scoring                  │
│                                    │
│  ┌──────────────────┐              │
│  │ tool_accuracy     │ 集合匹配     │
│  │ (0-40)           │ 缺/多工具扣分 │
│  └──────────────────┘              │
│  ┌──────────────────┐              │
│  │ answer_correctness│ 词汇重叠率   │
│  │ (0-40)           │ ×40           │
│  └──────────────────┘              │
│  ┌──────────────────┐              │
│  │ reasoning_coherence│ 重复检测    │
│  │ (0-20)           │ 迭代计数     │
│  └──────────────────┘              │
└──────────────┬─────────────────────┘
               │
               ▼ (when API key available)
┌────────────────────────────────────┐
│      Stage 2: LLM-as-Judge        │ ← 可选，需要 API Key
│      Refinement                    │
│                                    │
│  发送 Trace + 预评分结果给 LLM       │
│  LLM 输出多维度的精细分数和理由       │
│  可以是配或调整预评分                  │
└────────────────────────────────────┘
               │
               ▼
        返回结构化评分结果
```

**评分维度与权重**:

| 维度 | 权重 | 规则评分方法 | LLM 精化 |
|------|------|-------------|----------|
| tool_accuracy | 40分 | `calc_tool_accuracy()` 集合精确匹配，每个缺失/多余工具扣 10 分 | LLM 分析参数正确性和调用上下文 |
| answer_correctness | 40分 | 词汇重叠率 × 40，基于期望输出与提取答案的单词交集比 | LLM 语义级判断，处理同义词和改写 |
| reasoning_coherence | 20分 | 重复思考检测(每次 -4)，迭代过多(超 8 次扣分) | LLM 评估逻辑连贯性和推理质量 |

**混合模式行为**:

- 有 API Key → `_llm_refine()` 发送预评分 + Trace 给 LLM，LLM 返回精化后的 JSON 评分
- 无 API Key → 跳过 LLM 阶段，直接返回规则评分，设置 `mode: "rule_only"`
- LLM 调用失败 → 回退到规则评分，记录错误信息

**缓存机制**:
- 使用 `dict` 实现简单 LRU 缓存（max 128 条）
- 基于输入参数的 SHA256 哈希作为缓存键
- 相同输入直接返回缓存结果，避免重复 LLM 调用

**输出 `dict` 结构**:
```python
{
    "scores": {"tool_accuracy": 40.0, "answer_correctness": 32.0, "reasoning_coherence": 16.0},
    "total": 88.0,
    "reason": "[Rule-based] Tool: ... | Answer: ... | Coherence: ...",
    "token_cost": 0,
    "mode": "rule_only" | "hybrid",
    "details": {
        "tool_accuracy": {"score": 40.0, "reason": "...", "expected": [...], "actual": [...]},
        "answer_correctness": {"score": 32.0, "reason": "...", "extracted_answer": "..."},
        "reasoning_coherence": {"score": 16.0, "reason": "...", "iteration_count": 3}
    },
    "step_analysis": [
        {"index": 0, "has_thought": true, "has_action": true, "action_name": "web_search", "is_expected": true},
        {"index": 1, "has_thought": true, "has_action": true, "action_name": "final_answer", "is_expected": None}
    ]
}
```

### 3.5 异步任务层

**文件**: `backend/app/core/celery_app/tasks.py`

**三个 Celery 任务**:

```python
# 子任务 1: 执行测试用例
@celery_app.task(bind=True, max_retries=2)
def run_single_test_suite(self, test_suite_id: str, agent_config: dict) -> dict

# 子任务 2: 评分 Trace
@celery_app.task(bind=True, max_retries=2)
def run_judge_evaluation(self, trace_id: str, expected_output: str, expected_tools: list) -> dict

# 编排任务: 完整流水线
@celery_app.task(bind=True)
def run_full_evaluation(self, task_id: str) -> dict
```

**评测流水线**:

```
run_full_evaluation(task_id)
  ├── 1. 加载 Task + TestSuites
  ├── 2. 状态 → running
  ├── 3. Celery group: 并行执行所有 suites
  │     └── run_single_test_suite
  │           ├── OpenAIReActRunner.run()
  │           └── 持久化 Trace
  ├── 4. Celery group: 并行评分所有 traces
  │     └── run_judge_evaluation
  │           ├── LLMJudge.evaluate()
  │           └── 持久化 MetricScore
  ├── 5. 汇总：平均分 + 各维度分数
  └── 6. 状态 → completed | partial | failed
```

**错误隔离**: 每个子任务包裹 try/except，单个用例失败不影响其他用例执行。

## 7. 安全性设计

### 7.1 API Key 管理

- OpenAI API Key 通过环境变量 `OPENAI_API_KEY` 注入
- LLMJudge 自动检测 API Key：有则启用混合评分，无则纯规则评分
- 后端启动时不验证 API Key 有效性，避免启动失败

### 7.2 CORS 配置

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:5173"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

- 仅允许开发环境前端地址跨域访问
- 生产环境需替换为实际部署域名

### 7.3 输入验证

- 所有用户输入通过 Pydantic 模型验证
- SQLAlchemy ORM 防止 SQL 注入
- 禁止直接拼接 SQL 或 Shell 命令
- CSV/JSON 文件解析在前端完成，后端仅接收已验证的结构化数据

### 7.4 数据库安全

- 开发环境使用 SQLite，生产环境使用 PostgreSQL
- ORM 参数化查询防止注入
- 外键使用 CASCADE 删除确保数据完整性
