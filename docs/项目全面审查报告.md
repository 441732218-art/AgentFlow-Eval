## 1. 执行摘要

### 1.1 一句话结论

**AgentFlow-Eval 已从「Agent 评测工作台」演进为具备 Command Center 形态的 AI Agent 可观测 · 评测 · 诊断平台原型**：主业务闭环完整、企业横切能力齐全、多部署剖面可用，Intelligence UI 与 AOLS 日志骨架已落地；距离「生产级可观测 SaaS」仍差 **真实数据驱动闭环、诊断智能化、计费/多租户运营、版本与远程代码同步** 四条主线。

### 1.2 成熟度总览（审查评级）

| 维度 | 评级 | 说明 |
|------|------|------|
| 评测主链路（任务→执行→Trace→Judge→报告） | **A** | 端到端可跑通；Lite / Eager / Docker 三剖面 |
| Intelligence 驾驶舱 UI | **A-** | Dashboard / Trace / Diagnosis / Analytics / Monitoring 页面齐备；空数据与假数据边界仍需打磨 |
| 诊断引擎 | **B** | 启发式规则可用；非 LLM 根因分析，置信度偏展示向 |
| AOLS 结构化日志 | **B+** | Phase 2–4 完成：emit / 脱敏 / DB sink / API / Monitoring 接入；尚未全面驱动 Dashboard |
| 企业安全（RBAC / 审计 / 租户） | **A-** | 权限矩阵与前后端闭环；生产鉴权默认关闭场景仍多 |
| 计费 / SaaS 运营 | **C+** | 表结构 + mock Checkout；非真实支付与对账 |
| 插件与扩展 | **B+** | 动态加载、钩子、市场页、商业化元数据与权益门闩 |
| 多模态 | **B** | 上传/提取/评分路径具备；重度模型依赖可选 |
| 测试覆盖 | **A-** | 后端 unit ≈ 49 文件，场景 + E2E 有骨架；前端 Vitest/Playwright 偏薄 |
| 文档与软著 | **A** | 设计说明书、手册、专题文档、品牌、演示剧本齐全 |
| 远程同步 / 发布节奏 | **C** | 本地能力远超 `origin/main` 最后提交；需整理提交与版本发布 |

### 1.3 推荐使用形态（当前）

| 场景 | 建议 |
|------|------|
| 内部演示 / 评测验收 | ✅ Lite 或 Eager + seed 数据 |
| 私有化试用（全能力） | ✅ Docker Compose Private |
| 对外 SaaS 收费 | ⚠️ 仅骨架，需完成计费与运维闭环 |
| 替代 LangSmith / 全量 Agent APM | ❌ 暂不满足；按 §11 路线推进 |

## 2. 产品定位与目标用户

### 2.1 定位演进

| 阶段 | 定位 |
|------|------|
| **v0.1.0 基线** | 面向业务场景的 **Agent 自动化评测工作台** |
| **Unreleased（Intelligence）** | 企业级 **AI Agent Observability · Evaluation · Diagnosis** 控制中心（Cyber Command 主题） |
| **品牌名** | AgentFlow Intelligence；仓库路径仍为 AgentFlow-Eval |

**Tagline：** Agent Observability & Evaluation Platform  
**视觉对标气质：** LangSmith（Trace）× Datadog（监控驾驶舱）× 评测闭环（Judge / 报告）

### 2.2 解决的核心问题

| 痛点 | 平台应对 |
|------|----------|
| 人工抽检成本高 | 用例库 + 批量异步执行 |
| 只比最终答案 | Trace 步骤 + 工具调用链 + 多维 MetricScore |
| 过程不可追溯 | Trace 落库 + DAG + TraceID 透传 + AOLS 事件 |
| 故障定位靠肉眼日志 | Diagnosis 启发式 + Monitoring 日志流 + 错误拓扑 |
| 难工程化落地 | Alembic、RBAC、审计、部署剖面、CI |

### 2.3 目标用户

- 企业内部 Agent / 大模型应用质量与评测团队  
- 算法与 Prompt 工程人员（对比实验 / A/B）  
- 平台运维与 SRE（可观测 / 慢任务 / 配额）  
- 开源演示、技术分享、软著材料完整表达

## 3. 总体架构

### 3.1 逻辑架构

```
┌──────────────────────────────────────────────────────────────────┐
│  Frontend · AgentFlow Intelligence Center                         │
│  React 18 + Vite 5 + Ant Design 5 + ReactFlow + ECharts/Recharts │
│  Command: Dashboard · Trace · Diagnosis · Analytics · Monitoring   │
│  Evaluate: Tasks · Reports · Operate: Billing · Plugins · Settings │
└────────────────────────────┬─────────────────────────────────────┘
                             │ REST + WebSocket  /api/v1
┌────────────────────────────▼─────────────────────────────────────┐
│  Backend · FastAPI                                                │
│  tasks · traces · reports · diagnosis · dashboard · logs          │
│  experiments · ab · media · plugins · billing · observability     │
│  audit · settings · me · tools · ws                               │
└───────┬──────────────────────────────┬───────────────────────────┘
        │                              │
   ┌────▼────┐                  ┌──────▼──────┐
   │ Celery  │── Redis ──       │ PostgreSQL  │
   │ Worker  │  (Private)       │  or SQLite  │
   └────┬────┘                  └─────────────┘
        │
   AgentRunner → Trace → JudgeEngine → MetricScore / Report
        │              + AOLS emit → agent_logs / Prometheus
        └── Diagnosis heuristics / Business KPIs / Slow tasks
```

### 3.2 核心数据流

```
CSV/JSON 用例 → Task + TestSuite
       → 调度（Celery / Eager / Lite Queue）
       → AgentRunner（OpenAI ReAct | HTTP）
            → steps + tools + tokens
            → AOLS: agent.* / llm.* / tool.*
       → Trace 持久化
       → Judge（规则 + LLM + 人工复核）
       → MetricScore / Report 聚合
       → Dashboard 时序 · Diagnosis · Analytics · Monitoring 日志
```

### 3.3 部署剖面（Ports & Adapters）

| 剖面 | 队列 / 缓存 | 数据库 | 典型入口 |
|------|-------------|--------|----------|
| **Lite** | 进程内 Eager / Memory | SQLite | `scripts\start-lite.ps1` |
| **Eager 本地** | Celery always_eager | SQLite | uvicorn + vite |
| **Private** | Celery + Redis | PostgreSQL | Docker Compose |
| **SaaS 向** | 同上 + 计费/配额开关 | PostgreSQL | `BILLING_ENABLED` 等 |

可插拔端口：`TaskQueuePort` / `CachePort` / `EventBusPort` / `MeteringPort`。

## 7. 可观测与 AOLS 日志体系

### 7.1 三层可观测分工

| 层 | 机制 | 消费者 |
|----|------|--------|
| **指标 Metrics** | Prometheus counters/histograms | Grafana、`/metrics` |
| **业务聚合** | ORM 时序 / KPI / 慢任务 | Dashboard、Analytics |
| **结构化事件日志 AOLS** | structlog + LogEvent + DB sink | Monitoring、`/logs`、排障 |

### 7.2 AOLS 已完成阶段

| Phase | 内容 | 状态 |
|-------|------|------|
| 1 | 审计与架构设计（文档） | ✅ `observability-logging-audit.md` |
| 2 | structlog 硬化、脱敏、HTTP access、error_id | ✅ |
| 3 | Agent/LLM/Tool/Celery lifecycle 事件 | ✅ |
| 4 | `agent_logs` 表、DB sink、列表/统计 API、Monitoring 接入 | ✅ |

### 7.3 日志事件域（示意）

- `http.*` — 访问与失败  
- `agent.*` / `llm.*` / `tool.*` — 执行生命周期  
- `evaluation.*` — 评测编排  
- system / error_id — 异常对账  

### 7.4 审查结论

- **进步显著**：从「中文句子日志」升级为可查询事件流，并具备敏感字段脱敏与 Trace 关联字段。  
- **未闭环点**：  
  1. Dashboard 主路径仍以 **业务表聚合** 为主，未把 `agent_logs` 作为一等公民数据源；  
  2. 日志保留策略 / 采样 / 租户级隔离配额未产品化；  
  3. 与外部 Loki/ELK 的导出适配未做；  
  4. 前端 Monitoring 的实时推送（WS/SSE）可再增强。

## 11. 未来升级计划方案

> 原则：**先打通数据真相与演示可信度，再做智能化与商业化，最后做生态与规模。**  
> 每个 Phase 给出目标、范围、验收标准与预估工作量量级（人周为粗估）。

### Phase A — 真相源统一与演示硬化（1–2 人周）

**目标**：让驾驶舱、诊断、监控读到一致、可复现的数据；演示零尴尬。

| 工作项 | 说明 | 验收 |
|--------|------|------|
| A1 提交与版本 | 整理 Unreleased → 语义化 `0.2.0` 或 `0.1.x` 系列 tag | 干净 main + CHANGELOG 发布节 |
| A2 文档同步 | 更新功能模块清单、项目报告路由表、AOLS 文档状态 | docs 与代码一致 |
| A3 空态与 seed | 统一 EmptyState；seed 覆盖 Dashboard/Diagnosis 演示数据 | DEMO 剧本 15 分钟无卡壳 |
| A4 日志驱动面板（MVP） | Monitoring + Dashboard 至少 1–2 卡片直接查 `agent_logs` | 有评测后卡片有数 |
| A5 Command Center E2E | Playwright 覆盖五页冒烟 | CI 绿灯 |

### Phase B — 诊断智能化与评测深度（2–4 人周）

**目标**：从启发式升级为「可解释 + 可行动」的诊断与评测。

| 工作项 | 说明 | 验收 |
|--------|------|------|
| B1 LLM Diagnosis（可选开关） | 在规则结果上叠加 LLM 摘要与改进建议 | 输出结构化 suggestion + 引用 step |
| B2 规则可配置 | Issue 阈值/阈值 YAML 或 DB 配置 | 不改代码可调阈值 |
| B3 与 error_id / AOLS 关联 | 诊断页可跳转相关日志 | 深链可用 |
| B4 行业模板包 | 客服/RAG/工具调用 等 TestSuite 包 | 一键导入 |
| B5 基线 Baseline | 任务对比历史 run 的回归判定 | 报告出现 pass/fail vs baseline |
| B6 图表栈收敛 | 选定 ECharts 或 Recharts 为主 | 依赖减少、文档更新 |

### Phase C — 可观测生产化（2–3 人周）

**目标**：AOLS 达到「可接入企业日志栈」的水平。

| 工作项 | 说明 | 验收 |
|--------|------|------|
| C1 保留与采样策略 | 按 level/event 采样；TTL 清理任务 | 配置项 + 单测 |
| C2 导出适配 | stdout JSON 标准字段；可选 OTLP/Loki sink | 文档 + 示例 |
| C3 租户维度 | agent_logs 强制 tenant/actor 索引查询 | API 过滤齐全 |
| C4 告警钩子 | 错误率/慢任务阈值 → Webhook | 触发可测 |
| C5 Grafana 深化 | 面板绑定业务指标 + 日志示例 | 导入 JSON 可用 |

### Phase D — SaaS 与商业化（3–6 人周）

**目标**：可对外试用计费（若产品路线需要）。

| 工作项 | 说明 | 验收 |
|--------|------|------|
| D1 真实 Stripe（或其它） | 替换 mock checkout/webhook | 沙箱支付成功改订阅 |
| D2 配额硬门闩 UX | 402 提示、升级路径、账期 | 前后端一致 |
| D3 发票与账单导出 | PDF/CSV | 合规最小集 |
| D4 多租户加固 | 数据隔离审计、管理后台 | 渗透式测试清单 |
| D5 SSO | OIDC/SAML 至少一种 | 登录联调通过 |

### Phase E — 平台生态与规模（持续）

| 工作项 | 说明 |
|--------|------|
| E1 远程插件市场 + 签名校验 | 防供应链投毒 |
| E2 CI 门禁集成 | GitHub Action「跑评测套件」 |
| E3 水平扩展 | 按租户/队列优先级的 Worker 池 |
| E4 数据集版本管理 | suite 版本、可复现 run |
| E5 安全评测套件 | Prompt 注入、越权工具调用等 |
| E6 统一 Layout/API 清理 | 删除死代码与双轨 |

### 11.1 路线图示意

```
现在 ──► Phase A 真相与演示 ──► Phase B 智能诊断与评测深度
                │                        │
                └──► Phase C 可观测生产化 ─┴──► Phase D SaaS
                                              │
                                              └──► Phase E 生态与规模
```

### 11.2 不建议优先做的事

- 在真相源未统一前大规模重做 UI 视觉（收益低于数据可信）。  
- 未硬化 RBAC/租户前开放公网注册。  
- 同时引入过多 LLM 供应商抽象（现有 OpenAI 兼容已覆盖多数网关）。  
- 为「软著页数」继续膨胀无运行路径的代码。

## 12. 结论与建议优先级

### 12.1 结论

AgentFlow Intelligence（AgentFlow-Eval）当前是一套：

1. **业务闭环完整**的 Agent 评测平台；  
2. **UI 已具备控制中心气质**的可观测 / 诊断产品壳；  
3. **工程与文档资产远超 Demo** 的 monorepo；  
4. 本地 **Unreleased 能力显著领先远程 main**，需要发布治理。

适合：**私有化演示、内部评测回归、软著与开源展示**。  
需继续投资后才适合：**多租户收费 SaaS、替代专业 Agent APM**。

### 12.2 建议立即执行的 P0（按序）

1. **代码落库与推送**：按模块拆分提交，避免「能力在本地、远程仍是旧版」。  
2. **CHANGELOG 发布节 + 版本号对齐**（建议 Intelligence 合并为 `0.2.0`）。  
3. **Phase A**：seed + 空态 + 1–2 个日志驱动卡片 + Command 冒烟 E2E。  
4. **文档**：用本报告替换过时路由表认知；更新 `功能模块清单.md` 与 docs 索引。  
5. **生产路径演练**：`check-prod` + `post-deploy-verify` + DEMO 剧本全流程录像/截图归档。

### 12.3 成功指标（建议 1–2 个迭代后复评）

| 指标 | 目标 |
|------|------|
| 演示剧本完成时间 | ≤ 15 分钟，零手工 SQL |
| Dashboard 有数据率（seed 后） | 核心卡片 100% 非空 |
| 后端 unit | 保持 CI 全绿，关键路径不回退 |
| 诊断可用率 | 失败 Trace ≥80% 给出非 healthy 主因 |
| 主分支与 origin 差异 | 无长期堆积 Unreleased |
