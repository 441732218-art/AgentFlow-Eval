## 1. 先说结论（你最关心的）

| 问题 | 结论 |
|------|------|
| **开源 / 公开 GitHub 会不会导致不能申请软著？** | **一般不会。** 软著保护的是「著作权」，不是发明专利的「新颖性」。代码已经写出来，著作权依法自动产生；登记是确权与取证。**公开不等于不能登记。** |
| **MIT 协议会不会把著作权送给别人？** | **不会。** MIT 是许可他人使用，**著作权仍归权利人**（个人/单位）。你仍可以拿自己的软件去登记软著。 |
| **那还有没有风险？** | 有的是 **实务风险**，不是「一公开就废」：权属不清、版本与材料不一致、被抢注纠纷、单位/个人主体不符、把密钥与未定稿材料误公开等。 |
| **最稳妥的策略是什么？** | **先固定软著版本与材料 → 提交申请（或至少材料齐备锁定）→ 再大规模宣传/推公开仓库。** 今日本地已领先 origin、**尚未 push 全部 Intelligence 能力**，这是有利窗口。 |

## 2. 软著 vs 专利 vs 开源（别混）

| | 发明专利 | 软件著作权（软著） | 开源许可（如 MIT） |
|--|----------|-------------------|-------------------|
| 核心 | 技术方案、新颖性 | 代码与文档的「作品」表达 | 授权他人如何使用 |
| 是否要求未公开 | **通常要求**（新颖性） | **一般不因「已在 GitHub 公开」而驳回** | 与是否登记无必然冲突 |
| 与本项目 | 若未申请专利，公开影响小 | **主线** | 你已挂 MIT |

**一句话：**  
担心「公开了就不能申请」——那是 **专利思维**。软著场景下，更该担心 **权属、版本、材料一致性与主体**，而不是「绝对不能公开」。

## 4. 推荐策略（按稳妥程度）

### 策略 A — 最稳（推荐，若登记在即）

1. **冻结软著登记版本**（例如明确：登记版本 = **V2.0.0**，对应某次 tag 或某次打包目录，不要混用未完成的实验分支叙事）。  
2. **按该版本生成/核对**：  
   - 源程序鉴别材料（前后各 30 页或局方要求格式）  
   - 文档鉴别材料（设计说明书、用户手册等）  
   - 截图、运行环境说明  
3. **提交软著申请**（或至少交给代理并取得「已收件」类确认）。  
4. 再执行：  
   - `git push` 全量 Intelligence  
   - 对外宣传、发版说明、博客  
5. 公开仓库可继续 MIT；在 README 增加一句：  
   > 软件著作权登记版本为 V2.0.0；开源仓库持续迭代，版本号以 CHANGELOG 为准。

### 策略 B — 可并行（登记材料已齐、审核周期中）

- **可以**保持 GitHub 公开（尤其已公开过）。  
- **暂停**的是：用不稳定的半成品覆盖「正在登记的那一版」表述；或公开与权利人冲突的声明。  
- 审核期间改代码 **通常允许**（软著登记的是申请时的作品版本，不是禁止以后开发）。

### 策略 C — 暂缓公开（若尚完全未交件且极在意）

- 将仓库设为 **Private**，或暂不 `push` 本地 ahead 的 10 个 commit。  
- 本地开发、录屏、生成材料均可在私有环境完成。  
- 证书下来或申请号到手后再 Public + 宣传。

**结合你当前状态：** 本地 **ahead origin 约 10 commit 且未 push** → 非常适合走 **策略 A**：先锁 V2.0.0 材料并递交，再 push/发版。

## 5. 公开时「不要做」清单

| 不要 | 原因 |
|------|------|
| 把 **未提交申请** 的源程序鉴别 PDF/前后 30 页全文当宣传文案到处贴 | 无必要暴露鉴别页选择逻辑；材料以交局为准即可 |
| 权利人写成 A，仓库/LICENSE 写成 B，合同再写成 C | 权属争议是真风险 |
| 软著写「独立开发」，实际大量复制第三方且未披露 | 独创性与侵权风险（开源依赖本身通常 OK，注明即可） |
| 把客户数据、内网地址、API Key 推进公开库 | 安全与合规 |
| 以为「开源了软著就作废」而乱改软件全称/版本制造多套互斥叙事 | 增加审查与后续维权成本 |
| 在软著审中把软件全称改成完全另一个产品名并大肆宣传 | 名称与申请表不一致易出补正 |

## 6. 开源与软著可以同时成立的表述（示例）

可在 README 或文档中采用类似表述（按你真实主体改名）：

```text
本软件著作权归【权利人姓名/名称】所有。
已申请 / 登记计算机软件著作权：AgentFlow智能体评测管理平台软件 V2.0.0。
源代码以 MIT 协议开源，授权他人使用、修改与再分发，但不转让著作权。
开源仓库版本迭代见 CHANGELOG；软著登记以申请材料载明版本为准。
```

## 7. 建议你本周的动作顺序

```
① 确认权利人（个人 / 公司）与 LICENSE、GitHub 一致
② 锁定登记版本 V2.0.0（或你决定的版本）对应源码快照
③ 用 scripts 重新生成鉴别材料并人工抽查前后页
④ 提交软著（自行或代理）
⑤ 再决定：push origin / Public / 发 v0.2.0 / 对外介绍
```

**今日不要急着做的事（若软著优先）：**

- 不要为了「开源进度」强行 `git push --force` 或改写已公开历史。  
- 不要在未统一权利人名称时大规模转发仓库。  
- C12 发版可以等软著材料锁定后再做（发版与交件顺序：先交件更稳）。

## 8. 与「竞争定位 / 公开宣传」的关系

- **可以**在私有环境演示、给投资人/客户看、做软著录屏。  
- **对外公开技术博客 / 开源运营** 建议放在 **申请提交之后**。  
- 竞争分析、架构文章不替代软著材料；公开技术文章 **一般不阻却** 软著，但不要抄袭他人代码却主张独创。

## 9. 简要 FAQ

**Q：仓库已经 public 过了，还能申请吗？**  
A：一般可以。尽早以当前权属与固定版本提交即可。

**Q：MIT 开源后别人 fork 了，我的软著还有用吗？**  
A：有。软著证明你对作品的著作权；MIT 是你对公众的许可。维权与许可是不同层面。

**Q：开源了是否必须写「开源软件」？**  
A：按当地申请表填写；许多开源项目照常登记。以登记机关表格为准。

**Q：Intelligence 大更新要不要再登记？**  
A：重大版本可申请 **后续版本** 或新版本登记（看代理建议）；第一版先稳 V2.0.0。

**Q：本地 Phase A 的 commit 要不要先 push？**  
A：软著优先则 **可以暂缓 push**；交件锁定后再 push 更稳。
