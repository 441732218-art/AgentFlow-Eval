## 一、项目概述

### 1.1 一句话定位

**面向业务场景的 Agent 自动化评测工作台**：上传测试用例 → 异步执行 Agent → 可视化执行链路（DAG）→ LLM-as-Judge 多维打分 → 结构化报告。

### 1.2 背景与问题

随着大语言模型在客服、办公自动化、知识问答等企业场景中的深入应用，业务系统越来越依赖具备多步骤推理与工具调用能力的 Agent。传统质检方式存在明显短板：

| 痛点 | 说明 |
|------|------|
| 人工抽检成本高 | 覆盖面窄，难以形成常态化回归 |
| 只比最终答案 | 难以发现「用错工具、多调工具、中间推理偏离」等问题 |
| 过程不可追溯 | 缺少结构化 Trace，问题定位依赖日志肉眼排查 |
| 评测难工程化 | 多为零散脚本，无法与权限、审计、部署剖面统一落地 |

AgentFlow-Eval 将业务预期固化为 **TestSuite**，通过执行、轨迹记录与多维 **MetricScore** 形成可重复、可追溯、可评分的评测闭环，支持本地单机与容器化部署，**不强制依赖公网**即可完整运行。

### 1.3 项目目标

| 目标 | 达成方式 |
|------|----------|
| 端到端评测流水线 | 任务管理、用例导入、异步/同步执行、取消与归档 |
| 执行链路可视化 | Trace 落库 + ReactFlow DAG 展示 |
| 可信多维评分 | 规则指标 + LLM-as-Judge + 人工复核 |
| 可对接真实 Agent | OpenAI 兼容 ReAct Runner + HttpAgentRunner |
| 企业级可用 | API Key、RBAC、租户隔离、审计、限流、健康探针 |
| 多部署剖面 | Lite（零中间件）/ Eager / Private Docker 全栈 |
| 可扩展与可观测 | 插件系统、Ports & Adapters、Prometheus、Grafana |

### 1.4 适用场景

- 企业内部 Agent 质量回归与上线前验收
- 算法 / 评测人员的批量用例跑分与对比分析
- 开源演示、技术分享与产品原型
- 软件著作权登记材料展示（完整可运行系统）

## 二、总体架构

### 2.1 系统分层

```
┌─────────────────────────────────────────────────────────────┐
│  Frontend (React 18 + Vite + Ant Design + ReactFlow)        │
│  总览 / 任务 / 详情(DAG) / 报告 / 设置 / 插件 / 审计         │
└───────────────────────────┬─────────────────────────────────┘
                            │ REST + WebSocket  /api/v1
┌───────────────────────────▼─────────────────────────────────┐
│  Backend (FastAPI + Uvicorn)                                │
│  tasks · traces · reports · tools · experiments · ab        │
│  billing · plugins · media · observability · audit · ws     │
└───────┬─────────────────────────────┬───────────────────────┘
        │                             │
   ┌────▼────┐                 ┌──────▼──────┐
   │ Celery  │                 │ PostgreSQL  │
   │ Worker  │── Redis ──      │ 或 SQLite   │
   └────┬────┘                 └─────────────┘
        │
   AgentRunner → Trace → JudgeEngine → MetricScore / Report
```

### 2.2 核心数据流

```
用户上传 CSV/JSON 用例
        │
        ▼
  创建 Task + TestSuite
        │
        ▼
  调度执行（Celery / Eager）
  ├── Agent 多轮 ReAct / HTTP 调用
  │     └── 持久化 Trace（思考 / 动作 / 观察）
  └── Judge 混合评分
        ├── 规则预评分（工具准确率、答案重叠等）
        ├── LLM 精化评分
        └── 人工复核改分（可选）
        │
        ▼
  聚合报告 + 前端 DAG / 分数卡片可视化
```

### 2.3 部署剖面（Ports & Adapters）

| 剖面 | 特点 | 典型命令 / 配置 |
|------|------|-----------------|
| **Lite** | 零 Redis/Celery Worker，进程内队列 | `scripts\start-lite.ps1`，`DEPLOY_PROFILE=lite` |
| **Eager 本地** | SQLite + 同进程任务，最小依赖 | `CELERY_TASK_ALWAYS_EAGER=true` |
| **Private 全栈** | Postgres + Redis + Celery | Docker Compose / `scripts\docker-up.ps1` |
| **SaaS 向** | 计费配额、插件商业化等可选能力 | `BILLING_ENABLED` 等开关 |

可插拔端口：`TaskQueuePort` / `CachePort` / `EventBusPort` / `MeteringPort`，同一业务路径适配不同基础设施。

### 3.4 领域对象

| 对象 | 说明 |
|------|------|
| **Task** | 评测任务及状态机（排队 / 运行 / 完成 / 失败 / 归档等） |
| **TestSuite** | 单条用例：输入、期望输出、期望工具列表等 |
| **Trace** | 单次执行轨迹：步骤、Token、耗时 |
| **MetricScore** | 各维度分数与扣分说明 |

| **MediaAsset** | 多模态媒体资产 |
| **AuditLog / SlowTaskEvent** | 审计与慢任务事件 |
| **Billing** 相关表 | 套餐、订阅、用量、发票等（可选启用） |

## 四、技术选型

### 4.1 技术栈一览

| 层级 | 技术 | 用途 |
|------|------|------|
| 后端框架 | FastAPI 0.115 + Uvicorn | 异步 REST / WebSocket |
| ORM / 迁移 | SQLAlchemy 2.0 + Alembic | 数据模型与版本管理 |
| 任务队列 | Celery 5.4 + Redis 7.x | 异步评测流水线 |
| LLM | OpenAI 兼容 SDK | Agent 执行 + Judge 评分 |
| 数据库 | SQLite（开发/Lite）/ PostgreSQL（生产） | 持久化 |
| 前端 | React 18 + TypeScript + Vite 5 | SPA 工作台 |
| UI | Ant Design 5 + ReactFlow (@xyflow) | 组件与 DAG |
| 状态 / 数据 | Zustand + TanStack React Query | 客户端状态与服务端缓存 |
| 图表 / 导出 | Recharts、jsPDF 等 | 报告可视化与 PDF |
| 测试 | pytest、httpx、Vitest、Playwright | 后端单测/E2E、前端单测/E2E |
| 弹性 | tenacity 等 | 重试、超时、熔断配合自研层 |
| 容器 / 部署 | Docker Compose、Nginx、Railway/Vercel 配置 | 本地全栈与云前端托管 |

### 4.2 工程规模（约）

| 维度 | 规模 |
|------|------|
| 后端 Python 模块（`backend/app`） | ~150 个 `.py` 文件，约 630 KB |
| 前端 TS/TSX（`frontend/src`） | ~82 个文件，约 290 KB |
| 后端测试 | ~49 个测试相关 Python 文件（unit / e2e / scenarios） |
| API 端点模块 | 15+（tasks、traces、reports、experiments、ab、billing、plugins 等） |
| 数据库迁移 | 11 个版本脚本 |

*说明：统计不含 `node_modules`、`__pycache__` 等依赖与编译产物。*

### 4.3 核心设计亮点

1. **评测领域状态机**：任务生命周期与 pipeline 纯函数编排，便于单测与复用。  
2. **Runner / Judge 可插拔**：OpenAI ReAct 与 HTTP Runner 统一工厂；规则 + LLM 混合评分。  
3. **Ports & Adapters**：队列、缓存、事件、计量与部署剖面解耦。  
4. **企业横切能力**：鉴权、RBAC（ADMIN / MANAGER / REVIEWER / USER / GUEST）、租户、审计、安全响应头。  
5. **生产就绪**：settings_guard、check-prod、健康探针、限流、TraceID 全链路透传。  
6. **可观测闭环**：Prometheus 指标 + Grafana 仪表盘 JSON + 业务 KPI / 慢任务 API。

## 九、风险、限制与后续建议

### 9.1 当前限制

| 项 | 说明 |
|----|------|
| LLM 依赖 | 完整 Agent/Judge 能力需配置 OpenAI 兼容 API Key；无 Key 时依赖 mock/规则降级路径 |
| 计费 | Stripe Checkout 为 mock 骨架，默认无真实扣款 |
| 前端托管 | Vercel 仅适合前端；全栈仍依赖自托管 API 与数据层 |
| 版本标识 | 开源 CHANGELOG 为 v0.1.0，软著材料标注 V2.0.0，对外发布时建议统一版本叙事 |

### 9.2 后续可演进方向（建议）

1. **真实计费与多租户运营**：对接正式支付、发票与套餐策略。  
2. **更丰富的评测数据集与行业模板**：开箱即用的垂直场景套件。  
3. **评测结果与 CI 集成**：PR 门禁式 Agent 回归。  
4. **更大规模并发与队列策略**：按租户隔离 Worker、优先级队列。  
5. **统一版本发布节奏**：语义化版本 + 变更日志与软著版本对齐说明。

## 十、结论

AgentFlow-Eval 已形成 **可运行、可观测、可评分、可部署** 的 Agent 评测工作台：

- **业务闭环完整**：任务 → 用例 → 执行 → Trace → 评分 → 报告。  
- **工程能力扎实**：异步流水线、多部署剖面、迁移、CI、安全与审计。  
- **企业扩展到位**：RBAC、租户、实验 / A/B、多模态、插件与可选计费。  
- **文档与交付齐全**：设计说明、操作手册、部署清单、演示剧本与软著材料。

项目适合作为 **企业内部 Agent 质量回归平台**、**开源演示产品** 以及 **软件著作权完整表达** 使用。在完成生产配置校验与密钥治理后，可进入正式部署与持续迭代阶段。
