## 1. 项目总览

### 1.1 解决什么问题

企业在落地 LLM Agent 时，普遍卡在三类能力断裂：

| 断裂点 | 传统做法的不足 | AgentFlow 的回答 |
|--------|----------------|------------------|
| **质量无法批量化验收** | 人工抽检、只看最终答案 | 用例库 → 异步跑 Agent → 多维 Judge → 报告 |
| **过程不可见** | 散落日志、难还原工具链 | Trace 步骤落库 + DAG / Explorer |
| **故障难归因** | 靠肉眼翻日志 | Diagnosis 启发式 + AOLS 结构化事件 + Monitoring |

因此产品不是「又一个 Chat UI」，而是面向 **质量工程与运维可观测** 的 Agent 平台：  
**评测闭环是底座，Intelligence Command Center 是门面，可观测与诊断是差异化纵深。**

### 1.2 产品演进轨迹

```
评测工作台 (v0.1.0)
    → 企业横切（RBAC / 审计 / 租户 / 部署剖面）
    → 扩展能力（实验 / A/B / 多模态 / 插件 / 计费骨架）
    → AgentFlow Intelligence（驾驶舱五页 + Design System + AOLS）
```

**今天（本地已落库能力）可以认为是：**

> 一套 **演示级到私有化试用级** 的完整 monorepo：  
> 业务主链路可跑通，控制中心 UI 完整，工程与文档齐备；  
> 尚未达到替代云端专业 Agent APM / 成熟 SaaS 计费平台的程度。

### 1.3 今日工程结论（结合 Phase A 落库）

| 维度 | 结论 |
|------|------|
| 业务闭环 | **完整**：任务 → 用例 → 执行 → Trace → Judge → 报告 |
| 产品壳层 | **完整**：Dashboard / Trace / Diagnosis / Analytics / Monitoring |
| 工程化 | **扎实**：Alembic 至 012、三部署剖面、CI、大量后端单测、前端可 build |
| 文档与品牌 | **齐全**：审查报告、模块清单、专题文档、品牌规范、演示剧本 |
| 远程同步 | **待发版**：能力在本地领先 origin；需 C12 + push |
| 商业化 | **骨架**：Stripe mock、配额门闩，非生产支付 |

## 2. 功能结构（怎么拆解这套系统）

### 2.1 四层能力模型

```
┌─────────────────────────────────────────────────────────────┐
│  L4  产品面 · Intelligence Command Center                     │
│  驾驶舱 / Trace / 诊断 / 分析 / 监控 / 计费 / 插件 / 设置       │
├─────────────────────────────────────────────────────────────┤
│  L3  领域能力                                                 │
│  评测流水线 · Runner · Judge · 实验/A/B · 多模态 · 诊断引擎   │
├─────────────────────────────────────────────────────────────┤
│  L2  平台横切                                                 │
│  RBAC · 租户 · 审计 · 缓存 · 弹性 · 插件 · 计费 · Ports       │
├─────────────────────────────────────────────────────────────┤
│  L1  运行与可观测底座                                         │
│  FastAPI · Celery/Eager · DB · Redis · AOLS · Prometheus      │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 前端信息架构（用户感知的结构）

| 导航分组 | 页面 | 用户价值 |
|----------|------|----------|
| **Command** | 驾驶舱 | 健康度、趋势、流水线拓扑一览 |
| | Trace | 单次执行步骤 / DAG / 指标（LangSmith 气质） |
| | 故障诊断 | 循环调用、工具失败、Token/Prompt 漂移等 |
| | 分析 | 雷达、热力、模型对比 |
| | 监控 | 控制面 + AOLS 日志流 |
| **Evaluate** | 任务 / 创建 / 报告 | 经典评测工作台 |
| **Operate** | 用量计费 / 插件市场 | SaaS 与扩展运营入口 |
| **System** | 设置 | 配置、主题、鉴权提示 |

默认入口：`/` → `/dashboard`。

### 2.3 后端能力域（系统结构）

| 域 | 代表模块 | 关键 API / 机制 |
|----|----------|-----------------|
| 评测执行 | tasks, agent_runner, celery, pipeline | 执行 / 取消 / 归档；OpenAI ReAct + HTTP Runner |
| 轨迹与评分 | traces, judge_engine, reports | Judge / 人工复核 / 报告聚合 |
| 智能中枢 | dashboard, diagnosis, analytics 数据 | overview、时序、启发式诊断 |
| 可观测 | observability, logs, AOLS, metrics | KPI、慢任务、Prometheus、agent_logs |
| 科学实验 | experiments, ab | 多变体对比、在线分流与显著性 |
| 多模态 | media, multimodal | 上传 / 提取 / 规则+视觉评分 |
| 扩展 | plugins | 动态加载、钩子、市场、权益 |
| 商业化 | billing | 套餐、配额、Checkout mock |
| 安全合规 | security, rbac, tenancy, audit | API Key、五角色、审计、安全头 |
| 部署适配 | ports/adapters, profiles | Lite / Private / SaaS 剖面 |

### 2.4 核心数据对象

```
Task ──┬── TestSuite (用例)
       ├── Trace (执行轨迹 steps / tokens / latency)
       │      └── MetricScore (多维分数)
       ├── Experiment / ExperimentRun
       └── （可选）计费 Usage / Quota

横向：AuditLog · AgentLog(AOLS) · SlowTaskEvent · MediaAsset · AB*
```

### 2.5 部署结构（交付形态）

| 剖面 | 依赖 | 适用 |
|------|------|------|
| **Lite** | 仅 Python + Node，进程内队列 | 个人演示、零中间件 |
| **Eager** | SQLite + 同进程任务 | 本地开发 |
| **Private** | Postgres + Redis + Celery | 私有化全能力 |
| **SaaS 向** | 上 + 计费开关 | 未来多租户收费（未完成） |

**边界：** Vercel 仅适合托管前端；API / 队列 / DB 需自托管。

### 2.6 功能完整度速览（摘要）

| 能力带 | 完整度 | 说明 |
|--------|--------|------|
| 评测闭环 | ★★★★★ | 可演示、可私有化试用 |
| Command UI | ★★★★☆ | 页面齐，空态/联动可再打磨 |
| 诊断智能 | ★★★☆☆ | 规则启发式；非深度 LLM RCA |
| 结构化日志 AOLS | ★★★★☆ | 采集与查询可用；尚未统一驾驶舱真相源 |
| 企业安全 | ★★★★☆ | RBAC 闭环；SSO 未做 |
| 插件扩展 | ★★★★☆ | 本地市场够用 |
| 计费运营 | ★★☆☆☆ | Mock 骨架 |
| 云端替代 LangSmith | ★★☆☆☆ | 功能气质接近子集，深度与生态不足 |

## 3. 行业地图：我们在和谁比

### 3.1 赛道切分

Agent / LLM 相关平台大致分五条赛道（常有交叉）：

| 赛道 | 代表产品 | 核心卖点 |
|------|----------|----------|
| **A. LLM / Agent 可观测（Tracing）** | LangSmith、Langfuse、Arize Phoenix、Helicone | Trace、跨 run 调试、prompt 版本 |
| **B. 评测与基准（Evaluation）** | Promptfoo、Braintrust、RAGAS 生态、DeepEval | 数据集、评分器、回归、CI |
| **C. 传统 APM / 日志** | Datadog、Grafana、ELK | 通用可观测，非 Agent 语义 |
| **D. 模型实验平台** | W&B、MLflow、Hugging Face | 训练/实验管理为主 |
| **E. 云厂商 Agent 套件** | Azure AI、AWS Bedrock、GCP Vertex | 托管一体化、强绑定云 |

**AgentFlow 的站位：A ∩ B 的「可私有化一体化」中间带**  
— 既有评测工作台，又有 Command Center 式可观测/诊断壳，并强调 **私有化 / Lite 零中间件 / 中文文档与软著交付**。

### 3.2 竞争对照（能力维度）

| 维度 | LangSmith | Langfuse | Promptfoo | Braintrust | **AgentFlow** |
|------|-----------|----------|-----------|------------|---------------|
| 产品成熟度 | 高（商业标杆） | 高（开源+云） | 高（评测专注） | 高（评测+实验） | **中（完整原型）** |
| Agent Trace UI | 极强 | 强 | 弱/外部 | 中强 | **中强（自研 Explorer+DAG）** |
| 批量化业务评测 | 有 | 有 | **极强** | **极强** | **强（任务/用例/报告主链路）** |
| LLM-as-Judge | 有 | 有 | 有 | 有 | **有（规则+LLM+人工）** |
| 故障诊断中心 | 弱于「诊断产品化」 | 偏分析 | 弱 | 偏实验 | **有独立 Diagnosis 页（启发式）** |
| 驾驶舱 / KPI | 有 | 有 | 弱 | 有 | **有 Cyber Command 驾驶舱** |
| 私有化 / 离线 | 企业版/贵 | **友好** | CLI/本地友好 | 有 | **极友好（Lite/Docker/Eager）** |
| 中文与软著交付 | 弱 | 中 | 弱 | 弱 | **强** |
| 插件市场 | 生态大 | 集成多 | 适配器 | SDK | **自研插件系统（本地）** |
| 计费多租户 SaaS | 成熟 | 成熟 | 偏工具 | 成熟 | **骨架** |
| 开源协议 | 专有为主 | MIT 等 | MIT | 混合 | **MIT 全栈** |
| 与 LangChain 绑定 | 深 | 中 | 浅 | 中 | **浅（OpenAI 兼容 + HTTP Runner）** |

> 表意为产品方向对照，非第三方评测分数。竞品持续迭代，定位应定期复核。

### 3.3 相对位置（战略象限）

用两轴粗定位：

- **横轴：** 评测深度（左弱 → 右强）  
- **纵轴：** 运行时可观测深度（下弱 → 上强）

```
  可观测 ↑
         │  LangSmith · Langfuse
         │           · Phoenix
         │                    ★ AgentFlow（目标区：右上内侧）
         │         Braintrust
         │  Promptfoo · DeepEval
         └────────────────────────→ 评测
```

**当前真实位置：** 右上象限的 **内侧偏中**——  
评测主链路已站稳，可观测/诊断「有产品形态」但深度与生态仍弱于 LangSmith/Langfuse；  
相对 Promptfoo，更重 **带 UI 的一体化平台与私有化交付**，而非纯 CLI 评测器。

### 3.4 我们的差异化（可对外讲的三点）

1. **一体化 Control Center**  
   评测工作台 + Trace + 诊断 + 分析 + 监控同壳，而不是「CLI 评测 + 另购可观测」。

2. **部署光谱完整**  
   从零中间件 Lite 到 Docker 私有化，适合国内政企/内网/软著演示场景；不强迫上云。

3. **工程与交付资产厚**  
   Alembic、RBAC、审计、插件、Ports 适配、中文文档、演示剧本、软著材料——更像「可交接的平台工程」，而非 demo 脚本。

### 3.5 我们必须诚实的短板

| 短板 | 竞争含义 |
|------|----------|
| 诊断仍是启发式，非业界级 RCA | 难与「智能运维」叙事硬刚 |
| Trace 生态弱于 LangChain/LlamaIndex 一等公民集成 | 开发者心智仍站 LangSmith/Langfuse |
| 无大规模公开 benchmark / 社区数据集 | Promptfoo/开源评测社区更有声量 |
| 计费与多租户未生产化 | 难直接做商业 SaaS 对标 |
| 远程 main 曾长期落后本地 | 开源曝光与信任受影响（Phase A 已在本地收敛） |
| 前端 E2E、双图表栈等工程债 | 长期维护成本 |

### 3.6 目标客户与「赢法」

| 客户类型 | 是否优先 | 赢法 |
|----------|----------|------|
| 需要内网/私有化的企业评测团队 | **是** | Lite/Docker + 中文文档 + 评测闭环 |
| 软著/验收/课题演示 | **是** | 完整 UI + 材料 + 演示剧本 |
| 已深度绑定 LangChain 要极致 Trace | 否（二期） | 先做集成适配，不硬抢 |
| 只要 CI 里跑 prompt 回归 | 部分 | 后续补 CLI/GitHub Action 门禁 |
| 要全球 SaaS 替代 LangSmith | 否（中长期） | 需可观测深度 + 生态 + 稳定性 |

**建议对外定位一句话：**

> 「可私有化部署的 Agent **质量与可观测控制中心**——评测是主干，诊断与驾驶舱是差异化，不绑定单一 Agent 框架。」

## 5. 综合结论

### 5.1 项目是什么

AgentFlow Intelligence 是一个 **以 Agent 自动化评测为根基、向可观测与诊断控制中心演进** 的开源 monorepo 平台。  
它在「能跑、能看、能评、能部署、能演示」五件事上已经对齐；在「能替代云端标杆、能规模化收费、能社区生态」三件事上仍处于建设期。

### 5.2 功能结构一句话

**Command 看见运行态 → Evaluate 保障质量 → Operate 扩展与计量 → System 治理配置**，  
底下由统一后端领域服务与 AOLS/指标底座支撑。

### 5.3 竞争位置一句话

处于 **LangSmith/Langfuse（可观测）与 Promptfoo/Braintrust（评测）之间的一体化、私有化友好开源位**；  
当前是该象限里 **产品完整度不错的挑战者/完整原型**，尚未成为赛道领导者。

### 5.4 下一阶段若只做三件事

1. **发版与同步**（C12 + push）：让远程与能力一致，建立可信度。  
2. **数据真相统一**：让 Dashboard/Diagnosis 与 AOLS/业务表一致，演示零尴尬。  
3. **诊断与评测纵深二选一做透**：要么 LLM RCA + 基线回归，要么 CI 评测门禁——避免继续横向铺功能。
